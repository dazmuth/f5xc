Challenge Lab 2: Access AWS Instance via SSH

Estimated time for completion: 35-45 minutes

Objectives

At the end of this lab students will be able to:
  • Access AWS instance using ssh
  • Explore a few AWS ssh commands
        
      
Step
Open Windows Terminal in student workstation
Step
Start ubuntu
ubuntu
Step
Navigate to .ssh folder
cd .ssh
It is necessary to be inside the .ssh folder so that the id_rsa key can be read.
Step
Verify existence of id_rsa keys
ls
If id_rsa and id_rsa.pub do not exist in this folder:
  • Navigate to f5xc folder
    cd ~/f5xc
  • Update folder
    git pull
  • Return to .ssh folder
    cd ~/.ssh
Step
Ensure key is not publicly available
chmod 400 id_rsa
Step
Access Instance shell via ssh
  • Paste the ssh command as provided by the instructor, which will be unique for each student
    • It will look similar to this: ssh -i "id_rsa" centos@ec2-14-88-251-191.compute-1.amazonaws.com
  • Accept the connection
    yes
Step
Explore available commands
help
In training, do not run any of the configure commands or factory-reset

Notes on a few commands

At any time press q to escape from a command's results. Use Up and Down keys to navigate results.
  • clean-debug-info removes existing debug files
  • collect-debug-info creates debug files, which are needed by F5XC Support
      
    https://f5cloud.zendesk.com/hc/en-us/articles/4411517730967-How-to-collect-debug-info-for-cloud-and-on-prem-Customer-Edge-CE-site
      
  • The configure commands are generally not needed except for advanced use cases
  • exec has three sub-commands
    • exec ls + <SPACEBAR> displays the available file folders
    • exec ping + <IP address> performs a standard ping, but also allows pinging of Private IP addresses of other AWS instances (which would otherwise not be possible "outside" of AWS)
    • exec uptime displays uptime of entire AWS "node," which does not mean individual Kubernetes nodes, but the entire cluster
  • execcli -h displays dozens of useful commands to help debug the CE
    For each execcli sub-command, add flag -h to see the available flags and commands. For example:
    execcli journalctl -h
  • exit closes this shell
  • get-config displays current configuration, much of which is also displayed inside F5XC
  • health displays everything related to this instance in extreme detail
  • log vpm displays exhaustive cluster detail
  • Use soft-restart -h to display running services, then soft-restart + [service name] to reboot it
  • status -h displays available statuses to be viewed
  • volterra-status displays the status of running Kubernetes services

Reboot Node and View Logs

Rebooting node does not destroy any of the pods, but reboots the entire cluster. Use reboot to troubleshoot container issues and isolate any troublesome services or pods.
Reboot takes approximately 15 minutes.
Step
Review current status
  • Review current container status
    volterra-status
      
    It may take 10-15 seconds for the statuses to appear.
  • Go to the Online Boutique and generate some traffic
  • In F5XC, view vK8s Pods (Distributed Apps > Virtual K8s > [your vk8s] > Pods)
      
    Notice that the Node is up and marked green.
  • In the Terminal, reboot the node
    reboot
    The connection to the Instance is severed.
Step
View node status
  • Go to the Online Boutique and try to generate traffic
      
    A 503 or 500 error is displayed as the node reboots.
  • In F5XC, view vK8s Pods (Distributed Apps > Virtual K8s > [your vk8s] > Pods)
    Notice that Pods are running, but the Node containing the pods is unhealthy and marked red. The vK8s site and associated nodes may remain unhealthy (yellow or red) for 20-35 minutes as health checks and pod statuses are updated.
  • In the Terminal, reconnect to the Instance by pressing the Up key until the ssh command originally provided by the Instructor appears
  • View status of containers
    volterra-status
    Notice that some of the containers are Ready while others are not.
  • View Health of Node
    health
  • Press the Down key to scroll through results then q to exit
Step
View Pod Logs in F5XC
  • In F5XC, view vK8s Pods (Distributed Apps > Virtual K8s > [your vk8s] > Pods)
  • Locate the frontend pod and click the 3 dots under Action
  • Select Show Logs
  • In the drop-down, select wingman
  • Review messages related to node reboot
Step
Type exit when finished

Expected Results

Students have:
  • Accessed AWS instance using ssh
  • Explore a few AWS ssh commands
        
      

Review

  • Private id_rsa key must be loaded into the production workstation, or attached to the instance inside AWS. In this training, the deployment type has the VPC is created inside F5XC, which generates an AWS Instance, so it is not possible to attach a key inside AWS for an already-created Instance
  • This lab is meant to introduce commands, but not go into detail or demonstrate the power of many of them. In a production environment be cautious when running commands because edits made here immediately impact production application(s)
  • Rebooting the cluster may help identify container or pod issues because pods themselves are not rebooted
